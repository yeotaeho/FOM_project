# 시점 불변성(Viewpoint Invariance)과 `landmarks` 한계

> **대상:** domain1 추출·채점 담당자  
> **관련 코드:** `hub/services/extraction_service.py`, `pose_geometry.py`, `video_visualizer.py`  
> **마지막 업데이트:** 2026-05-20

---

## 1. 문제 정의

### 1.1 `landmarks`만으로 비교할 때의 한계

`landmarks`는 MediaPipe가 추정한 **33개 관절의 원본 위치**입니다.

| 항목 | 설명 |
|------|------|
| **좌표계** | `x`, `y`는 화면 기준 0~1, `z`는 hip 기준 상대 깊이 |
| **의미** | “이 프레임에서 관절이 **카메라 화면 어디**에 보였는가” |
| **전문가 vs 사용자 비교** | ❌ **부적합** — 같은 동작이라도 촬영 위치·앵글이 다르면 X, Y가 크게 달라짐 |

골반 중심 이동(Mid-Hip Translation)과 몸통 길이 스케일링(`normalized_landmarks`)을 적용해도, **카메라가 위/아래·측면·대각선**에서 찍히면 2D 화면에 투영되는 비율이 왜곡됩니다. 이를 컴퓨터 비전에서는 **시점 불변성(Viewpoint Invariance)** 문제라고 부릅니다.

```
[같은 팔 동작]
  정면 촬영  →  화면상 (x, y) 패턴 A
  측면 촬영  →  화면상 (x, y) 패턴 B   ← A ≠ B (동작은 같음)
```

### 1.2 현재 파이프라인에서의 역할 분담

| JSON 필드 | 시점·체형 보정 | 용도 |
|-----------|----------------|------|
| `landmarks` | 없음 (원본) | 시각화, Power·Rhythm 등 **화면/시간** 분석 |
| `normalized_landmarks` | 위치·키 (2D 중심) | 점 거리 비교 (Accuracy 보조) |
| `bone_vectors` | 정규화 좌표 기반 방향 | 코사인 유사도 비교 (Accuracy 권장) |
| `joint_angles` | **내부 각도 (스칼라)** | **시점에 가장 강건** — Accuracy 1순위 |

**원칙:** 전문가·사용자 **동작 유사도(Accuracy)** 채점에는 `landmarks` 단독 비교를 피하고, 아래 대안 또는 `joint_angles` / `bone_vectors`를 사용한다.

## 2. 대안 1 — 내부 관절 각도(Internal Joint Angles) 활용

### 2.1 개요

**알고리즘적 해결:** 공간상 절대 방향(Vector)이나 화면 좌표(점) 대신, **관절이 꺾인 각도(스칼라)** 를 비교한다.

- 카메라 앵글이 달라지면 `어깨 → 손목` **방향 벡터**는 다르게 보인다.
- 하지만 `어깨–팔꿈치–손목`이 이루는 **사이각**은 정면·측면에서도 (이상적으로) 동일하다.

### 2.2 구현 (현재 상태)

**이미 추출 파이프라인에 포함됨** (`pose_geometry.py` → 프레임 JSON `joint_angles`).

| 키 예시 | 의미 (∠ABC, B=꼭짓점) |
|---------|------------------------|
| `left_elbow` | 어깨–팔꿈치–손목 |
| `right_knee` | 골반–무릎–발목 |
| `left_shoulder` | 팔꿈치–어깨–골반 |
| … | 총 10개 관절 각 (도) |

계산은 **`normalized_landmarks`** 좌표에서 수행하므로, 체형·화면 위치 보정 이후의 3D 방향 각에 가깝다.

### 2.3 채점 시 권장 방식

```text
Accuracy_frame = mean( |angle_user[j] - angle_expert[j]| )  over joints j
→ 임계값·가중치 적용 후 0~100 점수화
```

- `bone_vectors` 코사인 유사도와 **병행**하면 방향·각도를 이중 검증 가능.
- `landmarks` X,Y 직접 비교는 Accuracy에 **사용하지 않음**.

### 2.4 장단점

| 장점 | 단점 |
|------|------|
| 구현 난이도 **낮음** (이미 추출됨) | 손가락·머리 등 미세 동작은 각도만으로 부족할 수 있음 |
| 1주 MVP에 **가장 현실적** | MediaPipe Z 노이즈가 큰 각도(어깨 등)는 오차 가능 |
| 시점 변화에 **가장 강건(Robust)** | |

## 3. 대안 2 — 3D 몸통 평면 정렬 (수학적 3D 회전 보정)

### 3.1 개요

MediaPipe는 `x`, `y`뿐 아니라 **상대 깊이 `z`** 를 제공한다. 이를 이용해 양쪽 어깨·양쪽 골반 4점으로 **몸통 평면(Torso Plane)** 을 만들고, 가슴이 바라보는 **법선 벡터**가 카메라 정면(Z축)과 평행이 되도록 33개 관절 전체에 **회전 행렬(Rotation Matrix)** 을 적용한다.

### 3.2 구현 절차 (설계)

1. **몸통 평면 정의**  
   - 점: `left_shoulder`, `right_shoulder`, `left_hip`, `right_hip`  
   - 평면 벡터: 예) `v1 = right_shoulder - left_shoulder`, `v2 = left_hip - left_shoulder`  
   - 법선: `n = normalize(cross(v1, v2))` → “가슴이 향하는 방향”

2. **목표 방향**  
   - 카메라 정면을 `(0, 0, 1)` 또는 MediaPipe 관례에 맞는 축으로 정의

3. **회전 행렬 계산**  
   - `R` = `n` → 목표 축으로 맞추는 최소 회전 (Rodrigues / `scipy.spatial.transform.Rotation`)

4. **전 관절 적용**  
   - 각 관절 `(x, y, z)`를 Mid-Hip 기준으로 옮긴 뒤 `R` 곱 → 이후 기존 Torso Length 스케일링

5. **출력 필드 (제안)**  
   - `view_aligned_landmarks` 또는 `normalized_landmarks` 파이프라인 **Step C**로 통합

### 3.3 기대 효과

- 사용자가 대각선에서 춤을 춰도, 서버 내부에서는 **정면에 선 것과 유사한 좌표계**로 보정.
- `normalized_landmarks`·`bone_vectors`의 **시점 편차 추가 감소**.

### 3.4 리스크 (MVP에서 주의)

| 리스크 | 설명 |
|--------|------|
| **Z축은 추정값** | 단일 카메라(모노) 기반 깊이는 오차가 크다 |
| **과도한 회전** | 잘못된 법선 → 관절 좌표 **붕괴** |
| **개발 비용** | NumPy 회전·테스트·엣지 케이스(옆으로 누움 등) — **1주 MVP에 부담** |

### 3.5 권장 시점

- **Phase 2 이후** Accuracy 오차가 “촬영 각도”로 반복 재현될 때 도입 검토.
- 도입 전 **대안 1 + 대안 3**으로 데이터 품질을 먼저 올리는 것이 안전.

## 4. 대안 3 — UI/UX 촬영 통제 (MVP 현실 전략)

### 4.1 개요

모든 변수를 백엔드 수식으로 잡기보다, **프론트엔드(앱)에서 촬영 조건을 통제**하여 `landmarks`·Z축 노이즈 자체를 줄인다.

### 4.2 구현 (프론트엔드)

| 요소 | 내용 |
|------|------|
| **가이드 실루엣** | 카메라 프리뷰 중앙에 반투명 전신 실루엣 오버레이 |
| **문구 가이드** | “휴대폰을 배꼽 높이에 **수직**으로 세우고, 전신이 실루엣에 들어오게 맞춰 주세요” |
| **선택 기능** | 전신 미포함 시 업로드 경고, 흔들림·가로 촬영 감지 시 재촬영 유도 |

### 4.3 효과

- 카메라 앵글 **물리적 편차 최소화** → 백엔드 `normalized_landmarks` / `joint_angles` 품질 향상.
- **개발 비용 대비 효과**가 1주 MVP에서 가장 크다.
- 백엔드는 **단순 채점 로직(대안 1)** 유지 가능.

### 4.4 백엔드 연동

- API는 현재와 동일 (`POST /video/extract`).
- 추후 메타데이터 확장 예: `capture_metadata: { "guide_used": true, "orientation": "portrait" }` → 품질 낮은 샘플 필터링.

## 6. MVP 권장 전략 (Hi-Six 합의안)

1. **Accuracy 채점:** `joint_angles` 1순위, `bone_vectors` 코사인 2순위.
